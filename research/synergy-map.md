## Introduction

The comparative analysis (`research/comparative-analysis.md`) identified three meta-patterns across nine dimensions: Infrastructure vs Psychology, Complexity vs Portability, and Breadth vs Depth. This document translates those patterns into engineering decisions. Section 1 maps specific mechanisms where GSD and Superpowers produce combined value greater than either alone. Section 2 identifies where the two systems are genuinely incompatible — not just different, but architecturally conflicting. Section 3 proposes concrete resolution strategies for each conflict, referencing existing decisions D001 (Claude Code native) and D002 (dependencies allowed). Section 4 synthesizes everything into a unified methodology direction with 5–7 design principles that constrain M002's architecture.

Every entry in this document references specific mechanisms from the source analyses. Vague categories like "verification" or "planning" are replaced with named components — GSD's `gsd-verifier` agent, Superpowers' gate function in `verification-before-completion`, the academic three-tier memory model from Codified Context Infrastructure. If a mechanism is not specific enough to name, it does not belong in this document.

The twelve synergies cluster into three categories aligned with the comparative analysis's meta-patterns:

## 2. Conflict Points Analysis

Where synergies show complementary mechanisms, conflicts reveal genuinely incompatible design philosophies. A difference is "GSD uses TypeScript; Superpowers uses Markdown." A conflict is where choosing one approach structurally precludes the other — where the gain from one system's approach requires sacrificing something the other system's approach provides.

### Conflict 1: Dependency Policy

**GSD's position:** Dependencies are fundamental infrastructure. `gsd-tools.cjs` with 19 domain modules provides lockfile-based state management, atomic operations, path traversal prevention, and template rendering. The TypeScript SDK (16,623 lines) enables programmatic execution. These are not optional luxuries — they provide enforcement guarantees that prose directives cannot replicate. Direct Write/Edit to STATE.md is explicitly prohibited (anti-pattern rule #15) because it bypasses lockfile-based mutual exclusion, field validation, and consistent formatting.

**Superpowers' position:** Zero dependencies by design. `package.json` contains only `name`, `version`, `type`, and `main`. The contributor guidelines enforce this: "PRs that add optional or required dependencies on third-party projects will not be accepted." The system's portability — any platform that reads Markdown can use it — is a direct consequence of zero dependencies.

**Why this is a real conflict:** GSD's enforcement guarantees require code execution. Superpowers' portability requires no code execution. You cannot simultaneously require a Node.js runtime for state management and be deployable on any platform that reads Markdown. The constraint is structural: lockfile-based mutual exclusion, JSON parsing with prototype pollution prevention, and YAML frontmatter CRUD operations are programmatic capabilities that prose cannot replicate.

**What's lost by choosing GSD's side:** Platform portability. Superpowers runs on any AI coding agent that can read Markdown files — no installation, no build step, no runtime requirements. Adopting dependencies means the methodology only works on platforms that can execute Node.js (or whatever runtime the tools use). This narrows the addressable market and increases setup friction.

**What's lost by choosing Superpowers' side:** Enforcement guarantees. Without programmatic state management, the methodology cannot prevent concurrent state corruption, detect prompt injection patterns, enforce state transition rules, or provide atomic operations. Behavioral directives can tell an agent not to corrupt state, but they cannot structurally prevent it. The comparative analysis showed that GSD's programmatic verification catches failure modes that behavioral enforcement misses — stubs, empty handlers, hardcoded values that the agent doesn't recognize as problems.

**Stakes:** This conflict determines whether the unified methodology can run autonomously (GSD's enforcement enables unattended execution) or requires human oversight as a safety net (Superpowers' behavioral approach works well with an active human partner). D002 resolved this in favor of dependencies, but the tradeoff must be explicitly acknowledged in M002's architecture.

### Conflict 2: Execution Model

**GSD's position:** Multi-agent orchestration with 21 specialized agents. The orchestrator spawns a fresh agent per function (research, planning, execution, verification), with each agent getting a clean context window. Wave-based parallelism executes independent tasks simultaneously. Model profiles assign different capability tiers to different agent roles.

**Superpowers' position:** Single-agent skill loading within one context window. The agent loads relevant skills at task boundaries and follows their behavioral directives sequentially. The `subagent-driven-development` skill provides optional multi-agent execution, but coordination is behavioral — the skill tells the agent how to dispatch subagents — not programmatic.

**Why this is a real conflict:** Multi-agent orchestration requires infrastructure that single-agent skill loading structurally cannot provide: a spawn/collect cycle, inter-agent state passing, wave dependency analysis, parallel commit safety (lockfile-based mutual exclusion, `--no-verify` commits to avoid build lock contention), and model resolution per agent role. Building this infrastructure contradicts Superpowers' minimalism. But abandoning multi-agent orchestration means losing context isolation (each agent starts clean, eliminating accumulated confusion), parallelism (wave-based execution maximizes throughput on independent tasks), and role specialization (dedicated verifier agents that distrust executor claims by design).

**What's lost by choosing GSD's side:** Context continuity. In a single-agent model, the coordinator retains the full conversation history — it knows what was discussed, what decisions were made, what the human cares about. In a multi-agent model, each spawned agent starts from a constructed prompt, losing the nuance of the original conversation. Interactive development suffers because the agent executing the code doesn't have the context of the design discussion.

**What's lost by choosing Superpowers' side:** Isolation, parallelism, and structural separation of concerns. Without dedicated verification agents that distrust executor claims, verification becomes self-assessment — the same agent that wrote the code checks its own work. OpenHands' separate benchmarks repository and the academic finding that independent verification is essential (SASE's speed-vs-trust gap) both argue that self-verification is insufficient for autonomous execution.

**Stakes:** This conflict determines the methodology's scalability. Single-agent models work well for focused interactive development (1–3 file changes). Multi-agent models are essential for complex multi-phase projects where isolation and parallelism provide real throughput and quality benefits. The resolution must support both modes.

### Conflict 3: State Philosophy

**GSD's position:** Persistent state machine. All state lives in `.planning/` (now `.gsd/`) as human-readable Markdown and JSON. STATE.md tracks current position, decisions, blockers, and metrics. Phase directories contain CONTEXT.md, RESEARCH.md, PLAN.md, SUMMARY.md. Session continuity is handled through pause/resume with `continue-here.md` and structured HANDOFF.json. The state model enables: cross-session memory, progress tracking, regression detection (running prior phases' test suites), and project-level observability.

**Superpowers' position:** Stateless behavioral directives. No persistent state file, no context propagation chain, no session continuity protocol. Skills are loaded fresh each session. The only state-like artifact is the spec document written during brainstorming, which is a project artifact, not methodology state. The system's simplicity is a direct consequence of statelessness — no state to maintain, no state to corrupt, no state inconsistency bugs. GSD v1.32 specifically added STATE.md consistency gates to address state corruption bugs, illustrating the maintenance cost of persistent state.

**Why this is a real conflict:** Persistent state enables cross-session continuity but introduces state management complexity (corruption, staleness, concurrency). Statelessness eliminates that complexity but makes cross-session continuity impossible — every session starts from zero, losing all accumulated context. You cannot simultaneously have zero state overhead and cross-session project memory.

**What's lost by choosing GSD's side:** Simplicity. State management is a real engineering burden — GSD's lockfile-based mutual exclusion, stale lock detection, spin-wait with jitter, and STATE.md consistency gates are all solutions to problems that statelessness never creates. Every state mutation is a potential corruption point. Every state file is a maintenance obligation. The Codified Context Infrastructure paper documents that stale context causes agents to generate conflicting code — state that falls behind reality is actively harmful.

**What's lost by choosing Superpowers' side:** Project continuity. Without persistent state, the methodology cannot track what was done across sessions, cannot detect regressions against prior work, cannot resume interrupted tasks, and cannot provide project-level progress visibility. For any project spanning multiple sessions — which is most real-world software development — statelessness means repeating discovery work, re-making decisions, and losing the accumulated understanding that makes later work more efficient.

**Stakes:** This conflict determines whether the methodology can handle multi-session projects (state required) or is limited to single-session tasks (stateless is fine). Since D001 targets Claude Code — a tool developers use for multi-session projects — persistent state is architecturally necessary. But the state system should be designed with Superpowers' zero-overhead philosophy as a design pressure, aiming for the minimum viable state rather than the comprehensive state model GSD currently maintains.

### Conflict 4: Distribution Model

**GSD's position:** npm package with CLI tools, SDK, hook scripts, 60 workflows, 42 templates, 25 references, and a ~3,000-line installer that transforms content for 10 runtimes. Installation requires Node.js, creates multiple directories, and modifies host platform configuration files.

**Superpowers' position:** npm package with zero dependencies. 14 skill files, 1 agent, 3 deprecated commands, 1 session-start hook script, and platform manifests. Installation is trivial — clone or npm install, and the system works. No build step, no runtime requirements beyond what the host platform already provides.

**Why this is a real conflict:** GSD's rich distribution enables powerful capabilities (multi-agent orchestration, programmatic verification, state management) but creates installation friction and maintenance overhead. Superpowers' minimal distribution enables trivial deployment but cannot deliver capabilities that require runtime code. The installer's transformation layer (tool name mapping, hook event adaptation, path convention handling) addresses multi-runtime support — but D001 constrains the unified methodology to Claude Code, potentially eliminating the need for this complexity.

**What's lost by choosing GSD's side:** Ease of adoption. Superpowers' zero-friction installation means a developer can start using it in under a minute. GSD's installation requires understanding Node.js, configuring paths, and potentially debugging installer issues. The barrier to entry directly affects adoption rate.

**What's lost by choosing Superpowers' side:** Capability depth. Without CLI tools, the methodology cannot provide atomic state operations, template rendering, programmatic verification, or any runtime enforcement. The behavioral-only approach works for experienced developers who internalize the methodology, but provides no safety net for new users or autonomous execution.

**Stakes:** Since D001 constrains to Claude Code and D002 allows dependencies, the distribution model should optimize for Claude Code's native plugin/extension mechanisms rather than multi-runtime portability. This resolves much of the conflict — the methodology can use dependencies (D002) within a single platform's distribution model (D001) without needing multi-runtime transformation.

## 3. Resolution Strategies

Each resolution addresses a specific conflict from Section 2 with a concrete architectural decision. These resolutions are designed to become design decisions in M002.

### Resolution 1: Layered Dependency Architecture (Conflict 1: Dependency Policy)

**Decision:** The unified methodology adopts a two-layer architecture: a **behavioral layer** (Markdown skill files with iron laws, rationalization prevention, gate functions — zero dependencies) and an **infrastructure layer** (CLI tools, hook scripts, state management — Node.js dependencies). The behavioral layer is the minimum viable methodology — it works standalone on any platform that reads Markdown. The infrastructure layer adds enforcement guarantees, state management, and programmatic verification — it requires Node.js but provides capabilities behavioral directives cannot replicate.

**Mechanism:** Every behavioral directive (iron law, gate function, rationalization prevention table) lives in a skill file that has no runtime dependency. Every enforcement mechanism (lockfile-based state management, stub detection, prompt injection scanning) lives in a CLI tool or hook script that requires Node.js. The behavioral layer defines what should happen; the infrastructure layer ensures it does happen. A developer using only the behavioral layer gets Superpowers-quality behavioral shaping. A developer adding the infrastructure layer gets GSD-quality enforcement on top.

**Tradeoff acknowledged (per D002):** The infrastructure layer makes the methodology dependent on Node.js. This is accepted because: (a) Claude Code already requires Node.js, so the dependency adds no new platform requirement; (b) GSD's enforcement guarantees (lockfile-based mutual exclusion, programmatic stub detection, atomic state operations) provide real safety that behavioral directives cannot replicate; (c) the behavioral layer remains independently deployable for platforms that cannot run Node.js. The zero-dependency philosophy from Superpowers survives as a design constraint on the behavioral layer, not on the entire methodology.

**M002 design implication:** All skill files must be self-contained Markdown with no runtime dependencies. All enforcement tools must degrade gracefully when unavailable (the behavioral layer still works). The infrastructure layer should be packaged as a Claude Code extension (per D001) rather than a standalone CLI.

### Resolution 2: Modal Execution with Shared Behavioral Protocol (Conflict 2: Execution Model)

**Decision:** The unified methodology supports two execution modes selected per-project or per-task based on complexity:

- **Interactive mode** (Superpowers-derived): Single agent with skill loading for focused development sessions. The agent loads relevant skills (brainstorming, writing-plans, executing-plans, verification-before-completion) and follows their behavioral protocols. Subagents are dispatched per Superpowers' `subagent-driven-development` skill when tasks benefit from fresh context. This is the default mode for interactive development.

- **Orchestrated mode** (GSD-derived): Multi-agent pipeline with fresh context per agent for complex multi-phase projects. The orchestrator spawns specialized agents (researcher, planner, executor, verifier) with clean context windows, coordinates wave-based parallel execution, and manages inter-agent handoffs. This mode activates for auto-mode execution on multi-slice milestones.

**Shared behavioral protocol:** Both modes use the same behavioral directives — iron laws, rationalization prevention tables, gate functions, verification requirements. In interactive mode, these are loaded as skills. In orchestrated mode, these are injected into each spawned agent's prompt. The behavioral content is identical; only the delivery mechanism changes.

**Tradeoff acknowledged:** Maintaining two execution modes adds complexity. This is accepted because: (a) the academic evidence (OpenDev, SASE) demonstrates that different task types need different orchestration approaches; (b) GSD's multi-agent model excels at autonomous multi-task execution while Superpowers' single-agent model excels at interactive development; (c) the shared behavioral protocol ensures quality consistency across modes even though the execution infrastructure differs.

**M002 design implication:** Agent definitions and skill files must be usable in both modes. A skill's behavioral content (iron laws, gate functions) must be extractable for prompt injection into spawned agents. The mode selection should be automatic based on project characteristics (number of tasks, autonomy level, session type) with manual override available.

### Resolution 3: Proportional State with Behavioral Anchoring (Conflict 3: State Philosophy)

**Decision:** The unified methodology implements **proportional state** — state complexity scales with project complexity, not as a fixed tax. Three state tiers:

- **Tier 0 (Stateless):** For single-session tasks and quick fixes. No persistent methodology state. Skills are loaded fresh; the session's context window is the only memory. This is Superpowers' model, preserved for the use cases where it excels.

- **Tier 1 (Lightweight):** For multi-session projects. Minimal persistent state: a project brief (what we're building), a knowledge register (what we've learned), and a decision log (what we've decided). No phase tracking, no progress metrics, no lockfile management. State files are human-editable Markdown.

- **Tier 2 (Full):** For complex multi-phase projects with autonomous execution. Full state management with phase tracking, progress metrics, session continuity (HANDOFF.json), lockfile-based mutual exclusion, and regression detection. This is GSD's model, preserved for the use cases where enforcement matters.

**Behavioral anchoring:** At all tiers, the behavioral directives (iron laws, verification gates, rationalization prevention) are active. Even in Tier 0 (stateless), the agent follows verification-before-completion and systematic debugging protocols. State management determines what is persisted across sessions; behavioral directives determine what happens within each session. The behaviors are constants; the state is variable.

**Tradeoff acknowledged:** Tier selection adds a decision point at project initialization. This is accepted because: (a) Superpowers' stateless approach genuinely excels for quick tasks, and forcing full state management on a 5-minute fix is hostile to the developer; (b) GSD's full state model is genuinely necessary for multi-phase autonomous execution, and attempting stateless multi-session projects leads to repeated work; (c) the tier boundary is detectable from project characteristics (task count, estimated duration, autonomy level).

**M002 design implication:** The state system must be designed so that Tier 0 requires zero setup, Tier 1 requires only creating 2–3 Markdown files, and Tier 2 is bootstrapped automatically when auto-mode launches a milestone. State tier can escalate mid-project (a quick fix that grows into a multi-session project upgrades from Tier 0 to Tier 1) but should not require re-doing prior work.

### Resolution 4: Claude Code Extension with Skill-First Surface (Conflict 4: Distribution Model)

**Decision:** The unified methodology distributes as a **Claude Code extension** (per D001) with a **skill-first user surface**. The developer's primary interaction is through skills — loading behavioral directives, invoking planning protocols, triggering verification. The extension's infrastructure (CLI tools, hooks, state management) operates behind the skills, providing enforcement and persistence without requiring the developer to interact with it directly.

**Mechanism:** Skills are the user-facing API. When a developer loads the brainstorming skill, the extension's infrastructure sets up the appropriate state tier, loads project context, and prepares the planning pipeline — all transparently. When the developer invokes verification, the behavioral gate function runs alongside programmatic stub detection — the developer sees a single verification result, not two separate systems.

**Tradeoff acknowledged:** Constraining to Claude Code (D001) means the methodology is not portable to other platforms without re-engineering. This is accepted because: (a) Anthropic's April 2026 policy blocks subscription in third-party harnesses, making multi-platform support moot for the immediate term; (b) Claude Code's extension API provides the hook points, tool permissions, and subagent spawning needed for both behavioral loading and programmatic enforcement; (c) the behavioral layer (Markdown skill files) remains independently portable — if D001 is revisited, the behavioral content can be repackaged for other platforms without rewriting.

**M002 design implication:** The extension manifest should register skills as the primary commands, with infrastructure services (state management, verification, context monitoring) running as background hook processes. Installation should be a single `claude extension install` command. The multi-runtime transformation layer from GSD's installer is not needed for M002 but should be designed as a potential future addition point.

## 4. Unified Methodology Direction

This section specifies which mechanisms survive, which are merged, which are dropped, and why — followed by 5–7 design principles that constrain M002's architecture. Every statement names specific mechanisms, not categories.

### Mechanisms That Survive from GSD

| Mechanism | Why It Survives |
|---|---|
| **Lockfile-based state management** (`state.cjs` with O_EXCL atomic creation, stale lock detection, spin-wait with jitter) | Concurrent state access safety is a programmatic property that behavioral directives cannot provide. Required for orchestrated mode where multiple agents may update state simultaneously. |
| **Stub detection patterns** (`verification-patterns.md` — comment-based stubs, empty implementations, hardcoded values, React-specific stubs, API-specific stubs) | Catches failure modes that agents do not self-recognize. The verifier's independence from executor claims (does NOT trust SUMMARY.md) is structurally important for autonomous execution. |

### Design Principles for M002

These seven principles constrain M002's architecture decisions. They are derived from the synergy/conflict analysis above, the comparative analysis's nine dimensions, and the academic/open-source validation.

**Principle 1: Behavioral layer is the foundation; infrastructure layer is the amplifier.**

Every quality-critical behavior (verification discipline, TDD compliance, root-cause-first debugging, planning before implementation) must be expressible as a standalone skill file with zero runtime dependencies. The infrastructure layer (hooks, CLI tools, state management) amplifies these behaviors with enforcement and persistence but is not required for them. This means: all iron laws, gate functions, and rationalization prevention tables live in Markdown skill files. All enforcement mechanisms (stub detection, state locking, prompt injection scanning) live in Node.js code. A developer using only the behavioral layer gets a useful methodology. A developer adding the infrastructure layer gets a robust one.

**Principle 2: Context is the primary engineering investment.**

Every architectural decision in M002 should be evaluated through the lens of context quality. What information reaches the agent, when, and in what form determines output quality more than any other factor. This means: the three-tier memory model (hot: system prompt + iron laws; warm: agent/skill definitions + task context; cold: reference documents + knowledge base) is an explicit architectural commitment, not an emergent property. Context content follows Superpowers' density principles (maximum impact per token, CSO-compliant descriptions). Context delivery follows GSD's managed approach (budget monitoring, threshold-based behavioral adjustment, adaptive enrichment for larger context windows). Drift detection (per Codified Context Infrastructure) prevents stale knowledge from causing incorrect output.

**Principle 3: Verification is dual-layered and non-negotiable.**

Every completion claim passes through both behavioral verification (gate function: IDENTIFY → RUN → READ → VERIFY → CLAIM) and programmatic verification (stub detection, 4-level artifact checking, independent codebase inspection). Neither layer can be disabled. The behavioral layer ensures agents actually run verification and read output. The programmatic layer catches failure modes agents do not self-recognize. This dual-layer approach addresses the "speed vs. trust gap" (SASE) from both directions — behavioral discipline narrows the gap at the agent reasoning level, programmatic checking narrows it at the codebase state level.

**Principle 4: State scales with project scope.**

The methodology does not impose a fixed state overhead. Tier 0 (stateless) for single-session tasks. Tier 1 (lightweight) for multi-session projects. Tier 2 (full orchestration) for autonomous multi-phase execution. Behavioral directives (iron laws, gate functions) are active at all tiers. State management (persistence, locking, tracking) scales up only when needed. Tier escalation is automatic when project characteristics exceed the current tier's capabilities. This preserves Superpowers' zero-overhead experience for quick tasks while providing GSD's full state machine for complex projects.

**Principle 5: Skills are tested behavioral specifications, not documentation.**

Every skill file is "code that shapes agent behavior" (per Superpowers' TDD-for-skills methodology). Skills are validated through adversarial pressure testing: run pressure scenarios without the skill loaded (observe natural agent behavior), load the skill (verify compliance), then close loopholes (refactor rationalization prevention tables). Each row in a rationalization prevention table represents a real evasion observed in testing, not a hypothetical. New skills enter the methodology through this quality gate; existing skills are refined through it when new evasion patterns are observed. Templates and reference documents follow the same discipline — they are load-bearing infrastructure (per Codified Context Infrastructure) with formal validation.

**Principle 6: Agent definitions combine capability contracts with behavioral specifications.**

Every agent in the unified methodology has two components: a **capability specification** (role, tools, model tier, completion markers, handoff schemas — derived from GSD's agent contracts) and a **behavioral specification** (iron laws, gate functions, rationalization prevention tables — derived from Superpowers' skills). Both are expressed declaratively in Markdown/YAML. Capability specifications define what the agent can do; behavioral specifications define how the agent should think while doing it. This addresses the gap each system has alone: GSD agents have clear capabilities but weak behavioral conditioning; Superpowers skills have strong behavioral conditioning but no capability contracts. Google ADK's declarative agent config and the cross-framework convergence on declarative agent definitions validate this approach.

**Principle 7: The methodology is opinionated about quality, flexible about process.**

The behavioral directives (verification discipline, TDD compliance, root-cause-first debugging, planning before implementation) are non-negotiable — they apply to every task regardless of complexity, execution mode, or state tier. The process infrastructure (execution mode, state tier, parallelism strategy, model selection) is flexible — it adapts to project characteristics, developer preferences, and task requirements. Iron laws are absolute; workflows are configurable. This means a developer can choose interactive or orchestrated mode, lightweight or full state, sequential or parallel execution — but they cannot skip verification, bypass the brainstorming gate for new projects, or claim completion without evidence. Quality is a constraint, not a feature toggle.

### Summary: What M002 Must Build

Based on the synergy map, conflict resolutions, and design principles above, M002's deliverables crystallize into five concrete components:

1. **Skill library** — Iron laws, gate functions, rationalization prevention tables, and behavioral protocols adapted from Superpowers, extended with GSD's domain coverage. Validated through TDD-for-skills. Zero runtime dependencies.

2. **Infrastructure extension** — Claude Code extension providing hooks (context monitoring, prompt injection scanning, behavioral reinforcement), CLI tools (state management, stub detection, template rendering), and the execution harness (agent spawning, wave coordination, handoff management). Node.js runtime.

3. **Agent definition format** — Declarative YAML/Markdown combining capability contracts (GSD-derived: role, tools, model tier, completion markers) with behavioral specifications (Superpowers-derived: iron laws, gate functions). Usable in both interactive mode (loaded as skills) and orchestrated mode (injected into agent prompts).

4. **Tiered state system** — Tier 0 (stateless), Tier 1 (project brief + knowledge register + decision log), Tier 2 (full phase tracking with lockfile-based concurrency). Automatic tier detection and escalation.

5. **Verification pipeline** — Dual-layer verification combining behavioral gates (Superpowers' IDENTIFY → RUN → READ → VERIFY → CLAIM) with programmatic checks (GSD's 4-level verification + stub detection). Independent verifier agent for orchestrated mode; self-verification with behavioral enforcement for interactive mode.

These five components, constrained by the seven design principles, define the architectural scope of M002.
